# FitMirror 需求分析

文档版本：v2.0  
最后更新：2026-03-31  
项目阶段：P0 已打通，当前进入真实训练闭环与文档基线阶段

## 一、当前阶段目标

当前阶段的核心目标不是继续扩散功能，而是把 MVP 从“可演示”推进到“可形成完整训练记录闭环”：

1. 用户输入目标并生成训练计划
2. 用户从计划页进入训练会话
3. 用户完成训练后自动写入记录
4. 热图和当天详情能反映真实训练结果

同时，要把项目文档收口到和代码一致，减少旧规划文档继续误导开发判断。

## 二、产品定位

FitMirror 是一个移动端优先的 AI 健身 MVP，当前验证重点是：

- 低门槛生成训练计划
- 低门槛完成一次训练
- 训练结果能够被记录和回看

换句话说，当前产品价值主张不是“全功能 AI 健身平台”，而是“能真正陪用户完成一次训练并留下记录的轻量闭环”。

## 三、当前需求优先级

### P0：已基本落地

- 注册 / 登录 / 鉴权
- AI 计划生成
- Dexie 本地数据层
- 训练热图
- 动作库

### P0 收口：当前阶段正在做

- 真实训练闭环
- 自动写入训练记录
- 现状文档基线
- 浏览器回归与测试稳定化

### P0+：后续增强

- 计划历史列表
- 热图周/月切换
- 趋势摘要
- 动作库收藏与最近查看

### P1 / P2：后续版本

- 动作识别
- 饮食推荐
- 社交能力
- 语音播报
- 数据导出
- 完整 PWA 离线与推送

## 四、当前用户主场景

### 1. 小白用户第一次完成训练

1. 注册并登录
2. 输入目标
3. 获得一份可执行计划
4. 点击开始训练
5. 训练完成后自动生成一条训练记录
6. 在热图中看到自己的第一次打卡

### 2. 用户回看训练结果

1. 打开热图页
2. 查看最近训练频率和总时长
3. 点击某一天查看当天训练详情

### 3. 用户借助动作库理解训练动作

1. 在计划页点击动作名
2. 跳转到动作库并带入筛选
3. 查看动作详情和注意事项

## 五、当前阶段的系统约束

- 不新增后端训练记录接口，继续以前端 Dexie 为记录主存储
- 不引入摄像头、姿态识别、语音播报
- 不把完整 PWA 能力作为当前阶段目标
- 保持当前设计语言，不开启新一轮大规模 UI 重构

## 六、验收标准

### 功能验收

- 计划页存在“开始训练”入口
- 训练完成后自动写入 `workout_records`
- `/records` 中能看到新记录
- 点击当天热图可查看本次训练详情

### 质量验收

- `npm run lint` 通过
- `npm run test:frontend` 通过
- `npm run test:backend` 通过
- 主链路浏览器回归通过

### 文档验收

- 文档中提到的页面、目录、数据表，都能在仓库中定位
- 文档中的“已实现 / 部分实现 / 未实现”状态与代码一致

## 七、当前不纳入本阶段目标的内容

- 动作识别实时纠正
- 复杂训练计时与阶段暂停策略
- 训练数据云同步
- 离线缓存与推送
- 社交与分享体系
