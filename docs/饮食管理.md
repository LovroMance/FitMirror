# FitMirror 饮食管理

文档版本：v0.1  
最后更新：2026-04-05

本文档用于收口 FitMirror 在 P1 阶段的饮食管理能力规划，重点包含“智能饮食推荐”与“食物营养信息提示”两部分。本文档描述的是实现方案与功能边界，不替代阶段进度记录；阶段推进情况统一写入 `docs/项目状态.md`。

## 一、目标定位

- 饮食管理作为训练计划之外的第二条辅助能力线，服务于减脂、增肌、保持等典型健身目标
- 第一阶段优先实现“基于用户目标的饮食知识检索与推荐生成”，让模型回答具备 FitMirror 自有知识约束，而不是直接输出通用大模型建议
- 同步补充“食物营养信息提示”，让用户能快速查看常见食物的营养构成、核心营养素与典型作用
- 当前阶段默认面向移动端 PWA 使用，不额外扩展桌面端专属流程

## 二、功能范围

### 1. 智能饮食推荐

- 用户输入健身目标，例如减脂、增肌、保持
- 用户可补充饮食偏好，例如高蛋白、少油、清淡
- 用户可补充忌口或限制条件，例如不吃辣、乳糖不耐受
- 系统基于内部饮食知识库先做检索，再结合检索结果调用大模型生成推荐
- 推荐结果优先输出“饮食原则 + 三餐建议 + 注意事项”，避免一开始扩展到过多维度

### 2. 食物营养信息提示

- 支持查看常见食物每 100g 的主要营养含量，例如蛋白质、碳水、脂肪、膳食纤维、热量
- 支持查看食物的主要营养价值说明，例如蓝莓富含花青素、可提供抗氧化相关营养价值
- 支持在推荐结果中穿插食物卡片或说明块，解释推荐依据
- 当前阶段优先覆盖高频健身相关食物，例如米饭、燕麦、鸡胸肉、鸡蛋、牛奶、蓝莓、西兰花、香蕉、三文鱼

## 三、推荐技术路线

### 1. 第一阶段不强依赖 LangChain

- 当前仓库已具备 `Node.js + Express + TypeScript + axios` 的后端基础能力
- 当前阶段的核心问题是“先把 FitMirror 自有饮食知识喂给模型”，而不是优先引入复杂编排框架
- 第一版可直接采用“静态知识库 + 轻量检索 + Prompt 注入 + LLM 生成”的简化 RAG 方案
- 在知识规模较小、推荐链路较短的前提下，自定义实现比直接引入 LangChain 更简单、可控、易调试

### 2. 第一阶段推荐栈

- 前端：`Vue 3 + TypeScript + Pinia + Element Plus`
- 后端：`Node.js + Express + TypeScript`
- 模型调用：沿用现有 LLM 调用方式，通过后端统一封装
- 知识库载体：后端本地静态 JSON
- 检索方式：标签过滤 + 关键词匹配 + 简单打分排序
- 结果生成：将检索结果拼入 Prompt 后调用模型生成最终建议

### 3. 后续可演进方向

- 当饮食知识量明显增大后，再评估 Embedding 与向量检索
- 当推荐链路出现多阶段检索、复杂 Prompt 模板管理、多模型切换需求后，再评估是否接入 LangChain
- 后续可扩展用户画像、历史饮食偏好、训练负荷联动等更强个性化能力

## 四、第一版 RAG 设计

### 1. 目标

- 让模型输出建立在 FitMirror 自有饮食知识之上
- 降低“通用大模型随意发挥”的比例
- 让推荐内容更贴近健身场景，而不是泛泛而谈

### 2. 基础流程

1. 用户提交目标、偏好和限制条件
2. 后端根据输入在饮食知识库中检索相关条目
3. 后端整理检索结果，生成结构化上下文
4. 后端将上下文与统一 Prompt 一起发送给模型
5. 模型输出推荐结果，前端按固定模块展示

### 3. 为什么静态库也能实现“不是大众 DeepSeek”

- 模型本身负责语言组织、归纳表达和自然输出
- FitMirror 饮食知识库负责提供产品自己的知识边界与建议口径
- 检索逻辑负责把当前问题最相关的知识送进模型上下文
- 只要 Prompt 明确要求模型优先依据检索内容回答，就能明显弱化裸模型的通用表达

## 五、知识库设计

### 1. 饮食建议知识条目

建议使用静态 JSON，单条结构可参考：

```json
{
  "id": "fat-loss-high-protein-breakfast-01",
  "type": "guideline",
  "title": "减脂期早餐优先保证蛋白质",
  "goalTags": ["fat_loss"],
  "preferenceTags": ["high_protein"],
  "mealTypes": ["breakfast"],
  "keywords": ["减脂", "早餐", "高蛋白", "饱腹感"],
  "summary": "减脂期早餐应优先保证蛋白质和适量复合碳水，避免高糖高油组合。",
  "content": "可优先选择鸡蛋、无糖酸奶、燕麦、全麦面包等组合，以提高饱腹感并稳定上午能量。",
  "priority": 8
}
```

字段建议：

- `id`：唯一标识
- `type`：条目类型，如 `guideline`、`meal_plan`、`food_fact`
- `title`：条目标题
- `goalTags`：适用目标标签
- `preferenceTags`：适用偏好标签
- `mealTypes`：适用餐次，如早餐、午餐、晚餐、加餐
- `keywords`：检索关键词
- `summary`：短摘要
- `content`：完整说明
- `priority`：当多条结果相近时的排序辅助值

### 2. 食物营养信息条目

建议将食物营养信息独立成另一份静态 JSON，单条结构可参考：

```json
{
  "id": "food-rice-cooked",
  "name": "熟米饭",
  "aliases": ["白米饭", "米饭"],
  "category": "staple",
  "unit": "100g",
  "nutritionPer100g": {
    "energyKcal": 116,
    "proteinG": 2.6,
    "carbohydrateG": 25.9,
    "fatG": 0.3,
    "fiberG": 0.3
  },
  "highlights": [
    "主要提供碳水化合物",
    "适合作为训练前后基础能量来源"
  ],
  "benefits": [
    "帮助补充日常活动与训练所需能量"
  ],
  "keywords": ["米饭", "碳水", "主食"]
}
```

补充说明：

- `nutritionPer100g` 用于回答“100g 米饭含多少克碳水”这类明确问题
- `highlights` 用于解释“这类食物主要提供什么”
- `benefits` 用于描述典型营养价值，例如蓝莓可提供花青素相关抗氧化价值
- 食物事实应尽量来源稳定、口径统一，后续若接入正式食物成分表，应统一替换为同一来源

## 六、检索设计

### 1. 第一版检索策略

- 先按 `goalTags`、`preferenceTags` 做基础过滤
- 再按 `keywords`、食物名称、别名做匹配
- 对命中的条目做简单分值计算
- 取前若干条结果进入模型上下文

### 2. 第一版为什么不急着上向量库

- 当前数据量可控，规则检索更直接
- 早期更需要验证知识条目结构是否合理，而不是先优化检索基础设施
- 规则检索更易排查召回原因，也便于快速迭代产品口径

### 3. 第二版再考虑的能力

- 用户问题改写
- Embedding + 向量召回
- 混合检索
- 引用依据展示

## 七、Prompt 约束建议

模型生成饮食推荐时，应明确要求：

- 优先依据系统提供的检索知识回答
- 若检索知识不足，允许做保守补充，但不能与知识条目冲突
- 输出时保持结构化，包含适用目标、三餐建议、食物替代建议、注意事项
- 避免输出疾病诊断、药物建议、极端节食方案
- 对食物营养信息优先引用系统提供的结构化数据，而不是自行估算

## 八、页面与交互建议

### 1. 饮食推荐入口页

- 目标选择
- 饮食偏好选择
- 忌口或补充说明输入
- 生成按钮

### 2. 推荐结果页

- 推荐原则摘要
- 早餐 / 午餐 / 晚餐 / 加餐建议
- 推荐食物说明
- 相关营养信息提示

### 3. 食物营养信息展示

- 支持显示“每 100g 营养含量”
- 支持显示“主要营养素”
- 支持显示“典型作用说明”
- 推荐结果页中的食物名称可触发营养信息卡片或详情弹层

## 九、第一阶段实现建议

推荐先按以下顺序推进：

1. 设计饮食知识库 JSON 结构
2. 设计食物营养信息 JSON 结构
3. 先整理一批高频数据，覆盖减脂、增肌、保持三类目标
4. 实现后端 `nutrition` 模块的检索与推荐接口
5. 前端补一个最小可用入口页与结果页
6. 最后再补更细的表现层和引用说明

## 十、边界与注意事项

- 当前阶段目标是“健身饮食建议”，不是医疗营养诊断
- 食物营养数据需保持来源口径一致，避免同类食物多份数据冲突
- 推荐结果应包含适度免责声明，提示用户若存在疾病、特殊体质或极端饮食需求，应咨询专业人士
- 在没有稳定数据源前，不宜一次性扩展太多食物，优先保证高频食物数据准确
