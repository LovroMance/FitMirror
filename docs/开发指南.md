# FitMirror 开发指南

> 本指南适用于所有前后端开发者，确保代码规范、简洁、易维护。请务必遵循以下原则和规范。

---

## 一、核心开发原则

- **SOLID**：面向对象五大原则，提升可维护性与扩展性。
- **KISS**（Keep It Simple, Stupid）：保持代码简单直接，拒绝过度设计。
- **YAGNI**（You Aren't Gonna Need It）：只实现当前需求，避免未用到的功能。
- **DRY**（Don't Repeat Yourself）：避免重复代码，提取通用逻辑。
- **单一职责**：每个模块/函数只做一件事。
- **最小可用**：优先实现最小可用功能，逐步迭代。
- **自动化校验**：所有提交必须通过 ESLint、Prettier、Stylelint、单元测试。

---

## 二、代码规范

- **命名规范**：变量、函数、类名语义清晰，采用小驼峰（js/ts）、大驼峰（类/组件）。
- **文件结构**：每个文件只包含单一功能，目录结构与文档一致。
- **注释简明**：仅在必要处添加注释，拒绝无意义注释。
- **类型安全**：前端全量 TypeScript，后端推荐 TypeScript。
- **函数简短**：单个函数不超过 40 行，复杂逻辑拆分。
- **无魔法数字**：常量集中管理。
- **无 console.log**：调试信息请用 logger 或移除。
- **无死代码**：定期清理未用代码。

---

## 三、前端开发约定

- **技术栈**：Vue3 + Vite + TypeScript + Pinia + Element Plus
- **本地存储**：统一用 Dexie.js 封装 IndexedDB
- **样式规范**：全局 CSS 变量，BEM 命名，禁止内联样式
- **组件复用**：通用组件放 components/common，业务组件放 components/business
- **布局复用**：页面公共壳层、认证区、导航区等结构，优先提取到 components/layout
- **设计令牌**：颜色、圆角、阴影、间距、字号、层级等视觉令牌统一沉淀到全局样式变量，禁止在多个页面重复硬编码
- **主题覆盖**：Element Plus 的主题变量与通用覆盖样式统一维护在全局主题文件中，禁止每个页面各自零散覆盖默认样式
- **页面职责**：views 负责页面组装、数据流转和页面专属样式；只有高重合、稳定的结构才下沉为组件
- **重复提取**：同一结构、样式或交互逻辑在多个页面稳定复用时，才提取为公共组件；局部且一次性的结构允许保留在页面内部，避免过度抽象
- **配置外置**：导航项、卡片 mock 数据、表单规则常量、视觉映射等优先放到 config、types 或 composables，不直接堆在页面文件中
- **状态管理**：全局用 Pinia，避免多余全局变量
- **API 调用**：统一封装于 src/api，返回 Promise，错误集中处理
- **表单校验**：所有表单需校验与错误提示
- **PWA 能力**：manifest.json、service worker、离线、推送

### 前端工程化补充要求

- 公共 UI 只在“高重合、稳定、明确复用”的前提下拆分为组件，避免为了组件化而组件化
- 页面内样式超过 200 行或出现大段重复时，必须评估是否拆分；但如果结构只服务单一页面，可优先保留在页面内部
- 新增页面前，先检查 `components/common`、`components/business`、`components/layout`、`assets/styles` 是否已有可复用资产
- 任何与品牌视觉强相关的颜色和层次，不允许直接依赖第三方组件库默认主题，必须显式对齐项目设计令牌

## 四、后端开发约定

- **技术栈**：Node.js + TypeScript + Express
- **数据库**：MySQL，ORM 使用 Prisma
- **接口风格**：RESTful，路径语义化，动词用 HTTP 方法表达
- **鉴权**：JWT，敏感接口需鉴权
- **日志**：统一用 logger，分级输出
- **错误处理**：全局中间件捕获，返回标准 JSON
- **单元测试**：Jest/Vitest 覆盖核心逻辑

---

## 五、代码提交与协作

- **分支规范**：main/dev/feature/bugfix，功能开发用 feature-xxx
- **提交信息**：遵循 Conventional Commits
- **代码检查**：提交前自动运行 lint、test
- **PR 审核**：至少 1 人 Code Review 后合并
- **文档同步**：功能变更需同步更新文档

---

## 六、API 响应格式规范

所有后端 API 必须返回统一结构，便于前后端协作与错误处理。

```typescript
// 成功响应
{
	"code": 0,
	"message": "success",
	"data": { ... }
}

// 错误响应
{
	"code": 40001,
	"message": "参数错误",
	"error": "email is required"
}
```

> 说明：
>
> - code=0 表示成功，非0为错误码（建议分业务模块分段）
> - message 为简要描述，error 为详细错误信息（可选）
> - data 为业务数据对象，错误时可省略

所有接口必须严格遵循此格式，前端统一处理 code/message/error。

## 八、常见反例（避免踩坑）

- 不要写大而全的函数/组件
- 不要提前设计未用到的功能
- 不要复制粘贴代码，提取为函数/组件
- 不要在业务代码中硬编码配置/常量
- 不要忽略类型检查和错误处理
