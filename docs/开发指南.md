# FitMirror 开发指南

文档版本：v2.0  
最后更新：2026-03-31

## 一、通用原则

- 保持实现简单直接，避免过度设计
- 只实现当前确定需求，避免预留空壳能力
- 保持单一职责，复杂逻辑主动拆分
- 优先复用已有模块，避免复制粘贴
- 代码变更必须同步更新相关文档

## 二、通用代码规范

- 前后端统一使用 TypeScript
- 命名语义清晰，组件使用大驼峰，变量和函数使用小驼峰
- 文件职责单一，避免一个文件混入多个领域逻辑
- 只在必要位置添加简短注释
- 禁止提交无用代码、无效常量和调试残留
- 新增常量优先沉淀到配置或类型定义，不在多处硬编码

## 三、前端规范

### 目录职责

- `views/` 负责页面组装和页面级状态流转
- `components/common/` 放稳定复用的通用组件
- `components/layout/` 放页面壳层和布局组件
- `api/` 统一封装 HTTP 请求
- `repositories/` 负责 Dexie 读写逻辑
- `assets/styles/` 维护全局样式与主题变量

### 组件与样式

- 只有高重合、稳定复用的结构才提取为公共组件
- 页面私有结构允许保留在页面内部，避免过度组件化
- 样式优先使用全局设计令牌和 CSS 变量
- 不允许在多个页面重复硬编码品牌色、圆角、间距和阴影
- Element Plus 覆盖样式统一放在全局主题文件中维护
- 禁止使用内联样式承载大段视觉规则

### 状态与数据

- 全局状态统一使用 Pinia
- 本地持久化统一通过 Dexie 与 repository 层访问
- 页面不直接堆叠大量数据转换逻辑，纯转换优先下沉到 `utils/`
- API 返回值统一在 `api/` 层做基础收敛，页面层只消费业务数据

## 四、后端规范

### 模块约束

- 路由层负责参数校验和响应输出
- service 层负责业务逻辑
- middleware 负责鉴权和统一错误处理
- 数据访问统一通过 Prisma

### 接口规范

- 使用 REST 风格路径
- 使用 HTTP 方法表达动作语义
- 受保护接口必须经过 JWT 鉴权
- API 响应统一采用标准包裹结构

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

```json
{
  "code": 40001,
  "message": "参数错误",
  "error": "detail"
}
```

## 五、测试与提交流程

- 提交前必须运行 lint
- 功能改动必须补充或更新对应测试
- 前端测试使用 Vitest
- 后端测试使用 Vitest
- 提交信息遵循 Conventional Commits

## 六、环境变量基线

### 后端

- `DATABASE_URL`
- `JWT_SECRET`
- `JWT_EXPIRES_IN`
- `PORT`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- `DEEPSEEK_MODEL`
- `DEEPSEEK_TIMEOUT_MS`

### 前端

- `VITE_API_BASE_URL`
