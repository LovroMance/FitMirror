# FitMirror 开发指南

文档版本：v2.6
最后更新：2026-04-07

本文档只定义长期开发规范、目录约束与交付要求，不记录阶段进度、实现状态和交接待办；这些动态内容统一记录在 `docs/项目状态.md`。

## 一、通用原则

- 保持实现简单直接，避免过度设计
- 只实现当前确定需求，避免预留空壳能力
- 保持单一职责，复杂逻辑主动拆分
- 优先复用已有模块，避免复制粘贴
- 代码变更必须同步更新相关文档
- 遵循 YAGNI：不为未确认的需求预写能力，不引入暂时用不到的抽象、字段、配置或流程
- 遵循 DRY：避免重复实现相同逻辑，已有能力可复用时优先抽取或复用，而不是复制代码
- 遵循 KISS：实现、命名、结构和交互都应保持清晰直接，优先选择最容易理解和维护的方案
- 遵循 SOLID：当模块开始变复杂时，用单一职责、依赖反转等原则整理结构；不要在简单问题上滥用设计模式

## 二、通用代码规范

- 前后端统一使用 TypeScript
- 命名语义清晰，组件使用大驼峰，变量和函数使用小驼峰
- 文件职责单一，避免一个文件混入多个领域逻辑
- 只在必要位置添加简短注释
- 禁止提交无用代码、无效常量和调试残留
- 新增常量优先沉淀到配置或类型定义，不在多处硬编码

## 三、前端规范

### 目录职责

- `views/` 负责页面组装和页面级状态流转
- `composables/` 负责页面级状态、异步流程和交互编排
- `components/common/` 放稳定复用的通用组件
- `components/layout/` 放页面壳层和布局组件
- `api/` 统一封装 HTTP 请求
- `repositories/` 负责 Dexie 读写逻辑
- `utils/` 负责纯函数、纯转换和无副作用逻辑
- `assets/styles/` 维护全局样式与主题变量

### 页面分层约束

- 页面必须按业务模块放入 `views/` 子目录，不允许所有页面平铺在 `views/` 根目录
- 页面文件统一使用 `XxxPage.vue` 命名
- `views/*Page.vue` 只负责页面结构、状态展示、事件绑定和少量页面入口承接
- 页面中禁止堆积大段异步流程、复杂表单提交、持久化调用和多段数据编排逻辑
- 页面需要的状态、提交、加载、弹窗、路由交互等逻辑，优先下沉到 `composables/<module>/useXxx.ts`
- `composables/` 按业务模块建子目录，例如 `auth/`、`plan/`、`workout/`、`home/`、`exercises/`
- `composables` 命名统一使用 `useXxx`
- `composables` 负责页面级逻辑复用，但不负责定义通用 UI 组件
- 若页面私有结构较长但不具备全局复用价值，应优先拆到页面所属模块内部，而不是直接上提到全局公共组件
- 只有跨模块稳定复用的结构，才放入 `components/common/` 或 `components/layout/`
- 当一个 `composable` 仍然过厚时，先继续拆成更小的 `composables` 或把纯逻辑下沉到 `utils/`；只有确实存在明显流程编排压力时，才允许新增中间层承接

### 页面、composables 与数据层边界

- `views/` 不直接承载复杂业务流程
- `composables/` 可以调用 `api/`、`repositories/`、`store/` 和 `router`
- `api/` 只负责请求和响应收敛，不负责页面状态管理
- `repositories/` 只负责本地持久化访问，不负责页面交互
- `utils/` 保持纯函数，不依赖 Vue、路由和持久化实现
- 页面里若出现大量 `watch`、异步请求、数据整理、错误处理并存，必须考虑拆到 `composables`
- 新功能开发默认先判断：这段逻辑属于页面 UI、页面状态、数据访问还是纯逻辑，禁止全部堆积在一个 `.vue` 文件里

### 组件与样式

- 只有高重合、稳定复用的结构才提取为公共组件
- 页面私有结构允许保留在页面内部，避免过度组件化
- 样式优先使用全局设计令牌和 CSS 变量
- 不允许在多个页面重复硬编码品牌色、圆角、间距和阴影
- Element Plus 覆盖样式统一放在全局主题文件中维护
- 禁止使用内联样式承载大段视觉规则

### 状态与数据

- 全局状态统一使用 Pinia
- 本地持久化统一通过 Dexie 与 repository 层访问
- 页面不直接堆叠大量数据转换逻辑，纯转换优先下沉到 `utils/`
- API 返回值统一在 `api/` 层做基础收敛，页面层只消费业务数据
- 页面状态优先通过模块内 `composables` 管理，而不是把所有交互都直接写在页面里

## 四、后端规范

### 模块约束

- 后端统一采用“模块化 + 模块内分层”结构，禁止回退到全局 `controllers/`、`services/`、`mappers/` 平铺堆积
- `modules/` 按业务模块拆分，例如 `auth/`、`plans/`、`workout-records/`
- `router` 只负责挂路由、中间件串联和入口转发
- `controller` 负责参数读取、调用 service、返回标准响应
- `service` 负责业务逻辑和流程编排
- `repository` 负责 Prisma / 数据访问
- `mapper` 负责 DTO / 响应结构转换
- `schema` 负责请求参数校验
- middleware 负责鉴权和统一错误处理
- 数据访问统一通过 Prisma
- 一个模块相关文件应尽量放在同一模块目录下，避免按技术类型散落到全局目录
- 新增后端能力时，优先判断它属于哪个业务模块，再决定是否需要新增 controller / service / repository / mapper / schema

### 接口规范

- 使用 REST 风格路径
- 使用 HTTP 方法表达动作语义
- 受保护接口必须经过 JWT 鉴权
- API 响应统一采用标准包裹结构

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

```json
{
  "code": 40001,
  "message": "参数错误",
  "error": "detail"
}
```

## 五、测试与提交流程

- 功能改动必须补充或更新对应测试
- 前端测试使用 Vitest
- 后端测试使用 Vitest
- 开发过程中的前置验证应按改动范围选择最小必要集合，优先尽早暴露问题，而不是一律在本地重复跑完整校验
- 若改动主要影响前端代码质量或类型边界，至少运行对应范围的 lint；若改动主要影响业务逻辑或数据流，至少运行对应范围的测试
- 若本地已知存在 git commit 钩子校验，可将其作为最终兜底；但不应完全跳过与本次改动直接相关的前置验证
- 每次完成更新后，都应补做一轮验证；验证范围应与本次改动匹配，优先选择最小必要集合，并在结果说明中写清本次实际采用的验证方式
- 浏览器验证不是默认必做项；当改动是否正确明显依赖真实页面表现时，例如布局、对齐、留白、遮挡、滚动、弹层、交互态、响应式或浏览器主路径结果，应主动进行浏览器检查
- 若通过代码检查、lint、测试或静态分析已足以确认结果，可不额外进行浏览器验证；但结果说明中仍应写清本次采用的验证方式
- 如果本次更新不适合通过浏览器直接验证，例如纯文档调整、纯后端内部逻辑或纯数据脚本改动，应在结果说明中明确写出原因，并补充对应的替代验证方式
- 提交信息遵循 Conventional Commits，默认使用中文描述，例如 `fix: 修复xxx`、`docs: 更新xxx`；除非用户明确要求，否则不使用英文提交信息

### 任务分级与最小流程

- 轻任务（仅问答、方案讨论、只读分析）默认不要求更新项目状态与提交 commit
- 标准任务（单模块或小范围改动）执行“修改 -> 最小必要验证 -> 更新项目状态 -> commit”
- 较大任务（跨模块、重构、长链路改动）在标准任务基础上增加分阶段验证
- 对纯文档改动，优先采用结构一致性检查与必要的差异复核；若不涉及页面表现，可不做浏览器验证，但需在结果说明中写明原因

## 六、推荐工具与代码规范

- 日期处理推荐 day.js（轻量、链式API）
- 强制使用 ESLint、Prettier、Stylelint 保证代码风格统一，建议在CI中集成自动检查
- Git 提交校验推荐 Husky（Git 钩子），配合 lint-staged 实现提交前自动格式化和校验
- BFF（Backend For Frontend）架构：前端通过专用BFF服务获取必要的远程数据，BFF仅做数据聚合与转发，不存储用户隐私数据
