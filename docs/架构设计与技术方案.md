# FitMirror 架构设计与技术方案

文档版本：v1.1  
最后更新：2026-04-03

本文档用于说明 FitMirror 当前阶段的架构设计、主要技术选型与数据边界。  
本文档以当前代码事实为准，不承载 P1/P2 的前瞻方案清单；若未来引入新能力，应在能力进入当前基线后再更新本文档。

## 一、整体架构概览

FitMirror 当前采用前后端分离的 Web 架构，前端以移动端 PWA 使用场景为主，基于 Vue 3 + Vite + TypeScript；后端作为 BFF 提供账号、计划同步、训练记录同步与计划生成相关 API。

当前数据模型采用“前端 Dexie 本地事实层 + 后端 MySQL 同步存储”的组合方式：

- 前端负责页面状态、训练流程、本地持久化与离线优先读取
- 后端负责账号体系、计划生成与训练计划 / 训练记录同步
- 动作库当前由前端静态 JSON 文件提供，不经过后端持久化

## 二、前端架构与技术选型

- 框架：Vue 3 + Vite + TypeScript
- 状态管理：Pinia
- 路由：Vue Router
- UI 组件库：Element Plus
- 网络请求：Axios
- 本地存储：Dexie.js（IndexedDB 封装）
- 日期处理：day.js
- 测试：Vitest
- 样式与校验：ESLint、Prettier、Stylelint、Husky、lint-staged

### 当前前端关键模块

- 账号体系：注册、登录、登录态恢复、路由守卫
- 训练计划：AI 生成、历史计划、计划编辑、计划同步
- 训练会话：按计划推进训练流程，完成后回写训练记录
- 训练记录：热图、趋势摘要、当天详情、记录同步
- 动作库：静态数据读取、筛选、详情、收藏与最近查看

## 三、后端架构与技术选型

- 运行时：Node.js + TypeScript
- Web 框架：Express.js
- 鉴权：JWT
- 数据访问：Prisma
- 数据库：MySQL
- 接口形态：RESTful API

### 当前后端职责

- 账号注册、登录、当前用户信息获取
- 训练计划生成能力与失败兜底
- 训练计划同步接口
- 训练记录同步接口

后端当前存储账号、训练计划、训练记录，不存动作库静态数据。

## 四、数据存储与同步边界

### 前端本地存储

- Dexie 作为本地事实层与页面优先读取层
- 本地持久化训练计划、训练记录、设置与动作偏好

### 后端存储

- MySQL 当前包含 User、Plan、WorkoutRecord 三类核心数据
- 训练计划与训练记录通过同步接口与前端本地层保持一致

### 当前不在后端持久化的内容

- 动作库静态数据
- 摄像头动作识别结果
- 饮食推荐、社交互动等未进入当前基线的能力数据

## 五、当前能力边界

以下方案不属于当前架构已落地事实：

- AI 动作识别 / PoseNet / 摄像头实时纠正
- 智能饮食推荐
- 社交激励 / WebSocket 实时互动
- 完整 PWA 离线缓存、推送与安装增强
- 数据导出、语音播报

当前 PWA 侧仅具备基础清单能力，尚未形成完整的离线与推送方案。

## 六、维护原则

- 当架构分层、核心技术栈或数据边界变化时，更新本文档
- 当只是阶段进度、视觉细节或交付记录变化时，不在本文档中记录
- 若本文档与代码或数据库事实冲突，以当前代码与 docs/数据库架构.md 为准，并同步修正文档
