在仓库 `d:\VS-code\FitMirror` 中工作时，先遵守以下规则：

1. 只要本次任务与代码、文档、配置、脚本、测试或评审有关，在开始分析、修改、实现、重构、测试或给出落地方案之前，先阅读 `docs/开发指南.md`。
2. 阅读 `docs/开发指南.md` 后，将其中的开发规范、目录约束、实现要求、协作方式和交付要求视为当前项目的主要工作指引，并在本次任务中持续遵守。
3. 每次开始实际修改前，必须查看 `docs/项目状态.md`，先理解当前阶段进度、已完成事项、技术边界、交接约定和下一步方向，避免重复建设或偏离当前阶段目标。
4. 每次完成修改后，必须再次查看并更新 `docs/项目状态.md`，简要记录本次改动内容、当前结果、遗留问题和建议的下一步事项。
5. 更新 `docs/项目状态.md` 时，`## 六、下次继续时建议先做的事`、`## 七、本次更新记录` 和 `## 八、下一步建议` 应采用覆盖式更新，只保留当前这一轮任务对应的内容，不在原有条目后持续追加累积。
6. 修改完成后的进度记录应保持简洁，只记录当前这次工作真正产生的结果，不写空泛总结，不复制规范类文档内容。
7. 每次完成更新后，都必须补做一轮验证；验证范围应与本次改动匹配，优先选择最小必要集合，并在结果说明中写清本次实际采用的验证方式。
8. 如果本次更新不适合通过浏览器直接验证，例如纯文档修改、纯后端内部逻辑或纯数据脚本调整，应在结果说明中明确写出原因，并补充对应的替代验证方式。
9. 每次完成修改并确认验证结果后，都必须进行一次 git commit；提交信息应准确概括本次改动，并遵循 Conventional Commits。若仓库已配置 git commit 钩子校验，可将其作为最终兜底，但不能替代与本次改动直接相关的前置验证。
10. 实施过程中默认遵循以下开发原则：
   - YAGNI：不要为未确认需求编写多余代码、预留空壳能力或提前扩展数据结构
   - DRY：不要重复实现已有逻辑，能复用就复用，确有必要时再抽取公共能力
   - KISS：保持实现、命名、结构和交互简单清晰，优先选择最直接可维护的方案
   - SOLID：当模块开始复杂时，再用单一职责、依赖反转等原则整理结构，不在简单问题上过度设计
11. 在无特殊情况时，不允许修改以下稳定基线文档：
   - `docs/开发指南.md`
   - `docs/功能说明.md`
   - `docs/视觉风格与页面原型设计规范.md`
   - `docs/需求分析.md`
12. 只有在以下特殊情况之一成立时，才允许修改上述稳定基线文档：
   - 用户明确要求修改该文档
   - 文档内容已经明显与当前代码事实或当前产品基线冲突，且如果不修正会持续误导后续开发
   - 当前任务本身就是文档基线收口、规范纠偏或产品基线调整
13. 即使出现特殊情况，需要修改上述稳定基线文档时，也应先明确说明修改理由，再进行修改；修改时必须保持最小改动原则，只修正必要内容，不顺手扩写无关部分。
14. 如果任务过程中需要多次进入不同模块，或在较长时间后继续同一任务，默认重新查看 `docs/开发指南.md` 和 `docs/项目状态.md`，确保理解没有偏差。
15. 每次任务都应有基本的执行节奏：
    - 修改前查看 `docs/开发指南.md`
    - 修改前查看 `docs/项目状态.md`
    - 完成实现或修改
    - 按变更类型完成最小必要前置验证，并在结果说明中写清本次实际采用的验证方式
    - 修改后更新 `docs/项目状态.md`
    - 完成一次准确的 git commit
    - 在记录中写清“本次修改了什么”和“下一步建议做什么”
16. 如果 `docs/开发指南.md`、`docs/项目状态.md` 不存在、无法读取、内容明显不完整，或不足以支持当前任务，应先说明情况，再基于现有上下文继续工作。
17. 如果用户的明确要求与上述规则冲突，可以按用户要求执行，但应先指出冲突点，再继续处理。
