在仓库 `d:\VS-code\FitMirror` 中工作时，遵守以下约定。

## 0. 目标

- 这些约定用于减少返工、保证可追溯、提升多人协作连续性。
- 默认在保证质量前提下，选择最小必要改动与最小必要验证。

## 1. 规则分层

### P0（必须）

1. 任务涉及代码、文档、配置、脚本、测试或评审时，在进入实现前先阅读 `docs/开发指南.md`。
2. 开始实际修改前必须查看 `docs/项目状态.md`，先确认当前阶段进度与边界。
3. 完成修改后必须更新 `docs/项目状态.md`，并保持记录简洁、可追溯。
4. 更新 `docs/项目状态.md` 时，`## 六、下次继续时建议先做的事`、`## 七、本次更新记录`、`## 八、下一步建议` 采用覆盖式更新，只保留本轮任务内容。
5. 每次完成修改后都要执行一轮与改动匹配的最小必要验证，并在结果说明中写清采用的验证方式。
6. 每次完成修改并确认验证结果后，必须执行一次 git commit；提交信息遵循 Conventional Commits，默认中文（如 `fix: 修复xxx`、`docs: 更新xxx`）。

### P1（默认遵守，可说明后调整）

1. 浏览器验证不是默认必做项，当大批量修改链路以及收口时，询问是否需要做浏览器验证。
2. 对于纯文档、纯后端内部逻辑、纯数据脚本等不适合浏览器验证的改动，需要在结果说明中写清原因，并给出替代验证方式。
3. 本地若存在 git commit 钩子校验，可作为最终兜底，但不能替代本次改动相关的前置验证。

### P2（建议）

1. 实施过程中默认遵循 YAGNI、DRY、KISS、SOLID，避免过度设计和重复实现。
2. 跨模块或间隔较久继续同一任务时，重新查看 `docs/开发指南.md` 与 `docs/项目状态.md` 以校准上下文。

## 2. 任务分流

1. 轻任务（仅问答、方案讨论、只读分析）：可不改文件，不强制更新 `docs/项目状态.md`，不强制 commit。
2. 标准任务（单模块或小范围改动）：执行 P0 全流程，采用最小必要验证。
3. 较大任务（跨模块、重构、长链路改动）：在标准任务基础上，增加分阶段验证与交付说明。

## 3. 稳定基线文档修改边界

默认不修改以下文档：

- `docs/开发指南.md`
- `docs/功能说明.md`
- `docs/视觉风格与页面原型设计规范.md`
- `docs/需求分析.md`

仅在以下情况可修改：

1. 用户明确要求修改；
2. 文档已与代码事实或产品基线冲突且会持续误导；
3. 当前任务就是文档基线收口、规范纠偏或产品基线调整。

满足条件时也应保持最小改动原则：只改必要内容，不扩写无关部分。

## 4. 默认执行节奏

1. 修改前：查看 `docs/开发指南.md`
2. 修改前：查看 `docs/项目状态.md`
3. 完成实现或修改
4. 执行最小必要验证，并记录验证方式
5. 修改后：更新 `docs/项目状态.md`
6. 完成一次准确 commit
7. 在记录中写清“本次改了什么”和“下一步建议做什么”

## 5. 异常与冲突处理

1. 若 `docs/开发指南.md` 或 `docs/项目状态.md` 缺失、无法读取、内容明显不完整，应先说明情况，再基于现有上下文推进。
2. 若用户明确要求与本规则冲突，可按用户要求执行，但应先指出冲突点再继续。
